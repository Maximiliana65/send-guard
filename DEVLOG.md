## 2026-07-16　設計スタート〜フェーズ1実装

### 話したこと・決めたこと
- そもそもの困りごと: AIとの会話でうっかりEnterや再試行ボタンを押して誤送信してしまう
- GitHubの既存拡張機能をそのまま入れるのは不安 → 自作する方針に
- Enterキーの挙動は「常に改行、送信は明示的な解除操作をした時だけ」に決定
- 送信ボタン・再試行ボタンの誤クリックも同じ仕組みでブロックすることに（Enterだけ対策しても根本解決にならないため）
- ロック解除は🔒アイコンのクリック＋ショートカット(Ctrl+Shift+U)の両方を用意
- 解除は「次の1回だけ」有効。送信したら自動でまた鍵がかかる（毎回慎重に、という方針）
- 対応サイトはまずClaude.aiのみ。動作確認後にChatGPT・Geminiへ拡大
- お楽しみ機能（一言コメント表示）は初期状態OFF。送信した瞬間に表示。コメントは`fun/comments.js`にただの配列として書くので、後から自分で気軽に追記できる形にした
- 保守性を重視し、「コア（共通ロジック）／サイトアダプター／お楽しみプラグイン」の3層構造に設計
- 将来的にGitHubで公開したいので、`_locales`を使った多言語対応の土台も最初から用意（今回は日本語＋英語）
- Gitはまずプライベートリポジトリで始める方針

### 作ったもの
- フェーズ1一式（`manifest.json`、`core/`、`adapters/claude.js`、`fun/`、`popup/`、アイコン、`_locales`）
- README / ROADMAP / CHANGELOG / この日誌
- 設計書（`docs/`）

### 次回への申し送り

- 動作確認しながら、バッジの表示位置やロック解除の感触を調整する可能性あり
- Git init〜GitHubへのプライベートリポジトリ作成はこれから

## 2026-07-18　ChatGPTでEnter送信がブロックされない不具合を調査・修正

### 報告内容
- ChatGPTで動作確認: 送信ボタン・再試行ボタンのロックは効いたが、Enterキーでは送信されてしまった
- ユーザー自身がChatGPTと一緒にF12のElementsパネルで調査し、入力欄が
  `<div id="prompt-textarea" class="ProseMirror" contenteditable="true" role="textbox">`
  であることを特定して報告してくれた（とても助かる調査だった）

### 原因(推定)
- 入力欄自体はセレクタに含まれており、検出はできていたはず
- 有力な原因は「タイミングの競争」: 拡張機能のコードが`document_idle`(ページ読み込み後半)で
  動き始めていたため、ChatGPT自身がEnterキーを処理する仕組みの方が一瞬早く反応してしまい、
  横取りする前に送信処理が進んでしまっていたと考えられる
- 送信ボタンのクリックの方はブロックできていたのは、おそらく別の仕組み(要素固有の処理)で
  実装されていて、タイミングの影響を受けにくかったため

### 対処
- `manifest.json`の`run_at`を`document_start`に変更(サイト自身のJSが動く前に監視を開始)
- 監視対象を`document`から一段外側の`window`に変更(より確実に先回りできるように)
- `document_start`では`<body>`がまだ無いことがあるため、バッジ表示だけ`DOMContentLoaded`まで
  遅延させるよう調整(キー・クリックの監視自体は即座に開始)
- バージョンを`0.2.0` → `0.2.1`(バグ修正のためPATCH)

### 次回への申し送り
- **この修正がChatGPTで実際に直っているか、再度確認が必要**
- ロック中にEnterで挿入される改行が、ProseMirror上で正しく表示されるかも要確認
  (もし改行が消えたり変な挙動になったら、挿入方法を再検討する)
- Geminiはまだ未確認のまま

### ちょっとしたつまずき
- フォルダ一括作成のコマンド(`mkdir -p .../{ja,en}`)がうまく展開されず、
  `_locales/ja` と `_locales/en` の中身(`messages.json`)が空のまま先に進んでしまっていた
- 最終チェックで発覚し、2つのファイルを作り直して解決済み
- 教訓: フォルダを複数まとめて作るコマンドを使ったときは、後でちゃんと中身ができているか一度見返すと安心

## 2026-07-18　ChatGPTのEnterガードをMAIN worldへ分離（v0.2.2）

### 再発確認
- `document_start` と `window` のcapture監視に変えた後も、実機ではChatGPTだけEnter送信を止められなかった
- 送信ボタンのロックは引き続き有効だったため、入力欄の判定ではなく、キーイベントの処理順が原因と判断した

### 設計変更
- **送信を止める処理だけ**を、ChatGPTページ本体と同じ `MAIN world` で実行する `core/chatgpt-page-guard.js` に分離した
- バッジ、ロック状態、設定、お楽しみ機能は従来どおり隔離されたcontent scriptの共通コアが担当する
- 両者の状態共有には、`<html>` の `data-send-guard-unlocked` 属性と小さなイベント通知を使う

### ねらい
- ChatGPT側のwindowイベントリスナーより先に、同じ実行コンテキストでEnterを止められる
- ChatGPT固有のイベント順序の問題を、Claude/Geminiの共通処理に持ち込まない
- 今後ほかのサービスで同じ問題が出た場合も、同じ構成の専用ページガードを追加できる

### 確認
- JavaScriptの構文チェックとmanifestの検証を実施
- 簡易イベントテストで、ロック中のEnterが送信されず改行処理になること、解除中のEnterが一度だけ通って再ロックされることを確認
- 実際のChatGPT画面での最終確認は、拡張機能を再読み込みしてから行う

## 2026-07-18　Geminiの解除済みEnter送信を修正（v0.2.4）

### 発生していたこと
- Geminiでは、ロックを解除してEnterを押しても送信されず、お楽しみ表示だけが出て再ロックされていた
- 一方で送信ボタンのクリックは正常だった

### 設計変更
- Geminiのキー監視を共通content scriptから、ページ本体と同じMAIN worldの専用ガードへ移動
- Enterを許可した直後には再ロックしない。Geminiが入力欄を空にしたことを監視して、送信開始を確認してから再ロックする

### 効果
- EnterイベントをGemini自身の処理にそのまま渡せる
- 送信されなかった場合はロック解除状態を保つため、お楽しみ表示だけが出る誤判定を防げる
